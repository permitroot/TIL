## 평가의 3단계

AI 시스템을 평가할 때는 세 가지 수준을 구분해야 한다.

| 단계 | 대상 | 주요 지표 |
|------|------|-----------|
| **모델 레벨** | 언어 모델링 능력 | 퍼플렉시티, 교차 엔트로피 |
| **태스크 레벨** | 특정 작업 수행 능력 | 기능적 정확성, 정답 유사도 |
| **시스템 레벨** | 실제 서비스 품질 | 사용자 만족도, 비용 효율 |

> 퍼플렉시티가 낮다고 해서 "사용자가 좋아하는 답"을 만든다는 보장은 없다.  
> 각 단계의 평가를 모두 설계해야 한다.

## 3.1 파운데이션 모델 평가의 어려움

기존 ML은 시험지와 정답지가 있었다.  
하지만 LLM은 **논술형 시험**이다 — 채점 기준을 직접 설계해야 한다.

### 3.1.1 모델이 똑똑해질수록 평가는 더 어려워진다

| 난이도 | 예시 | 평가 가능 여부 |
|--------|------|---------------|
| 쉬움 | 초등 수학 오류, 엉터리 요약 | 누구나 발견 가능 |
| 어려움 | 박사 수준 수학 풀이, 전문 분야 요약 | 전문가만 검증 가능 |

모델이 발전할수록 평가에 더 깊은 전문 지식과 더 많은 시간이 필요하다.

### 3.1.2 개방형 출력은 정답 기반 평가를 무너뜨린다

**기존 ML**: 출력 공간이 작고 고정됨 (A/B/C 중 하나 → 정답과 단순 비교)

**파운데이션 모델**: 정답이 여러 개 가능

```
Q: "서울의 장점은?"

가능한 정답:
- 교통이 편리하다
- IT 인프라가 좋다
- 문화시설이 많다
- 음식이 다양하다

→ 모두 정답이 될 수 있음
```

같은 입력에도 올바른 출력이 여러 개 → 모든 정답을 사전 열거 불가

### 3.1.3 블랙박스 모델

많은 파운데이션 모델이 폐쇄 소스이거나 내부 구조를 알 수 없다.

- 아키텍처, 학습 데이터, 학습 방식에 접근 불가
- 강점·약점, 편향, 학습 데이터 분포를 파악하기 어려움
- 관찰된 출력만으로 판단하는 **블랙박스 평가**에 의존
  - 실패 원인 진단이나 성능 격차 이해가 힘들어짐

### 3.1.4 벤치마크 포화와 데이터 오염

좋은 벤치마크는 모델이 발전해도 차별화 능력을 유지해야 한다.

| 벤치마크 | 출시 | 포화 시점 | 대체 |
|----------|------|----------|------|
| GLUE | 2018 | ~1년 | SuperGLUE (2019) |
| NaturalInstructions | 2021 | ~1년 | Super-NaturalInstructions (2022) |
| MMLU | 2020 | ~4년 | MMLU-Pro (2024) |

**데이터 오염 문제**
- 벤치마크 데이터가 사전 학습에 포함됐을 가능성
- 모델이 문제를 '기억'하는 것인지 '이해'하는 것인지 구분 불가
- 실제 일반화 능력 평가가 왜곡될 수 있음

### 3.1.5 평가 범위의 확장

- **전통 ML**: 단일 작업 전용 → "이 작업을 얼마나 잘하는가"만 평가
- **파운데이션 모델**: 코딩, 추론, 번역, 요약, 계획 등 수많은 작업 가능
  - 알려진 작업에서의 성능 측정뿐 아니라
  - 예상치 못했던 **창발적 능력(Emergent Abilities)** 발견도 평가의 일부
    - 특정 규모 이상에서 갑자기 나타나는 능력 (복잡한 추론, Chain-of-Thought 등)
    - 기존 지표로는 포착 어려움

> 평가자는 단순한 점수 기록자가 아니라 **능력의 탐험가**가 된다.

### 3.1.6 평가 생태계 현황

- 2023년 초: LLM 평가 관련 논문 월 2편 → 월 35편으로 폭발적 증가
- GitHub 상위 1,000개 AI 레포 중 50개 이상이 평가 도구에 집중
- **그럼에도 평가는 여전히 투자 부족**
  - DeepMind: 대부분의 노력이 새 알고리즘 개발에 집중, 평가 방법론 개선은 미흡
  - Anthropic: 평가 방법론과 기존 평가의 견고성 분석을 위한 투자 확대 촉구
- 많은 팀이 여전히 **'감(vibe check)'** 에 의존
  - 소수 프롬프트로 눈으로 확인 → 엄밀한 반복 개선, 회귀 탐지에는 부적합

## 3.2 언어 모델링 지표 이해하기

LLM은 확률 모델이다.  
목표: **다음 토큰의 확률 분포를 예측**  
→ 정보이론 기반 지표가 핵심

교차 엔트로피, 퍼플렉시티, BPC, BPB는 하나의 값과 필요한 정보가 있으면 나머지를 계산할 수 있다.  
텍스트뿐 아니라 이미지 토큰, 음성 토큰 등 **토큰 시퀀스를 생성하는 모든 모델**에 적용 가능하다.

### 3.2.1 엔트로피 (Entropy)

엔트로피는 **불확실성의 평균량**, 또는 '정보량의 평균'을 의미한다.

$$H(p) = -\sum_{x} p(x) \log p(x)$$

- $p(x)$: 사건 $x$가 일어날 확률
- $\log$를 밑 2로 쓰면 단위는 **비트(bit)**, 자연로그로 쓰면 단위는 **나츠(nats)**

#### 직관적 이해: 정사각형 위치를 표현하는 언어

| 언어 | 토큰 수 | 예시 | 필요 비트 | 엔트로피 |
|------|---------|------|-----------|---------|
| 언어 A | 2개 | `upper`, `lower` | $\log_2 2 = 1$ | **1 bit** |
| 언어 B | 4개 | `upper-left`, `upper-right`, `lower-left`, `lower-right` | $\log_2 4 = 2$ | **2 bit** |

언어 B의 엔트로피가 더 높음 → 각 토큰이 더 많은 정보를 담고 있음 → 다음 토큰 예측이 더 어려움

#### 핵심 성질

- 확률 분포가 **균등**할수록(동전 앞/뒷면이 50:50) 엔트로피가 **최대**
- 특정 결과에 **확신**이 높을수록(치우친 동전) 엔트로피가 **낮음**
- 언어 모델에서: 다음 단어 예측이 어려울수록 엔트로피가 높음

```
"오늘 날씨가 정말 ___"
→ 좋다 / 맑다 / 덥다 / 쌀쌀하다 ... → 선택지 多 → 높은 엔트로피

"대한민국의 수도는 ___"
→ 서울 (거의 확실) → 선택지 1개에 가까움 → 낮은 엔트로피
```

> **직관**: '모델이 얼마나 헷갈려 하는가?'

### 3.2.2 교차 엔트로피 (Cross Entropy)

교차 엔트로피는 **실제 데이터 분포와 모델 예측 분포의 차이**를 측정한다.

$$H(P, Q) = H(P) + D_{KL}(P \| Q) = -\sum_{x} P(x) \log Q(x)$$

| 기호 | 의미 |
|------|------|
| $P(x)$ | 실제(참) 분포 |
| $Q(x)$ | 모델이 예측한 분포 |
| $H(P)$ | 데이터 자체의 고유한 불확실성 |
| $D_{KL}(P \| Q)$ | 모델 분포 $Q$가 실제 분포 $P$를 얼마나 잘못 근사하는지 (KL 발산) |

#### 해석

- 모델이 실제 정답 토큰에 **높은 확률**을 부여할수록 교차 엔트로피가 낮아짐
- $Q = P$인 완벽한 모델이라면 $D_{KL} = 0$ → $H(P, Q) = H(P)$ (데이터 자체의 불확실성만 남음)
- **언어 모델 학습의 목표 함수 = 교차 엔트로피 최소화**

#### 왜 log를 사용하는가?

1. 작은 확률을 강조 (0에 가까운 확률에 매우 큰 패널티)
2. 수학적으로 미분 가능 → 역전파(backpropagation) 활용 가능
3. 정보량 단위(bit, nat)로 해석 가능

> **직관**: '이 모델이 이 데이터셋을 예측하기 얼마나 어려운가?'

### 3.2.3 문자당 비트(BPC)와 바이트당 비트(BPB)

모델마다 토큰화 방식(단어 단위, 서브워드, 문자 단위 등)이 달라 **'토큰당 비트'는 모델 간 비교가 불가능**하다.  
이를 표준화하기 위해 더 작은 단위로 정규화한다.

#### BPC (Bits per Character, 문자당 비트)

교차 엔트로피가 6 bits/token이고 토큰당 평균 2문자라면:

$$\text{BPC} = \frac{\text{교차 엔트로피 (bits/token)}}{\text{문자/token}} = \frac{6}{2} = 3 \text{ bits/character}$$

#### BPB (Bits per Byte, 바이트당 비트)

문자 자체도 인코딩 방식에 따라 크기가 다르다 (ASCII: 7bit, UTF-8: 8~32bit).  
BPC = 3이고 문자 하나가 평균 7/8 바이트라면:

$$\text{BPB} = \frac{\text{BPC}}{\text{bytes/character}} = \frac{3}{7/8} \approx 3.43 \text{ bits/byte}$$

**BPB = 3.43의 의미**: 원본 데이터(8 bits/byte)를 3.43 bits/byte로 압축 가능  
→ **좋은 언어 모델은 강력한 압축기(compressor)로 볼 수 있음**

#### 장점 정리

| 특징 | 설명 |
|------|------|
| 언어 독립적 비교 | 영어, 한국어, 중국어 모델을 동일 척도로 비교 |
| 공정한 모델 비교 | 토크나이저가 달라도 비교 가능 |
| 압축 효율 평가 | 모델의 정보 압축 능력 수치화 |

### 3.2.4 퍼플렉시티 (Perplexity)

퍼플렉시티는 교차 엔트로피의 지수 형태다.  
로그를 취한 값(교차 엔트로피)을 다시 지수로 돌려놓은 것.

$$\text{PPL}(P) = 2^{H(P)} \quad (\text{비트 사용 시})$$

$$\text{PPL}(P, Q) = 2^{H(P,Q)} \quad (\text{비트 사용 시})$$

$$\text{PPL}(P, Q) = e^{H(P,Q)} \quad (\text{나츠 사용 시, PyTorch/TensorFlow 기본값})$$

#### 직관: 유효 분기 계수(Effective Branching Factor)

| 교차 엔트로피 | 퍼플렉시티 | 의미 |
|--------------|-----------|------|
| 2 bits | $2^2 = 4$ | 평균적으로 4개 선택지 중 하나를 고르는 수준 |
| 1 bits | $2^1 = 2$ | 평균적으로 2개 선택지 중 하나 (동전 던지기 수준) |

> **직관**: '모델이 평균적으로 몇 개의 선택지 중에서 고르는가?'  
> 낮을수록 모델이 더 자신감 있게 예측 → **낮을수록 좋음**

#### 언어 모델로 퍼플렉시티 계산하기

토큰 시퀀스 $x_1, x_2, \ldots, x_n$이 주어졌을 때:

$$\text{PPL} = \left(\prod_{i=1}^{n} \frac{1}{P(x_i \mid x_1, \ldots, x_{i-1})}\right)^{1/n}$$

**계산 절차:**

1. 각 토큰 $x_i$의 로그 확률 $\log P(x_i \mid \text{context})$ 획득
2. 평균 로그 확률(avg_logprob) 계산:
   $$\text{avg\_logprob} = \frac{1}{n}\sum_{i=1}^{n} \log P(x_i \mid x_1, \ldots, x_{i-1})$$
3. 퍼플렉시티 계산:
   - 자연로그(ln) 사용 시: $\text{PPL} = e^{-\text{avg\_logprob}}$
   - $\log_2$ 사용 시: $\text{PPL} = 2^{-\text{avg\_logprob}}$

```python
import math
import numpy as np

# 실제 정답 분포 p(x)
p = np.array([0.7, 0.2, 0.1])

# 모델 예측 분포 q(x)
q = np.array([0.6, 0.3, 0.1])

cross_entropy = -np.sum(p * np.log2(q))
perplexity = 2 ** cross_entropy

print(f"Cross Entropy: {cross_entropy:.4f} bits")
print(f"Perplexity:    {perplexity:.4f}")
```

> 주의: 토큰 수준 확률(logprobs)에 접근할 수 있어야 하며, 일부 상용 API는 이를 제한적으로만 제공한다.

### 3.2.5 퍼플렉시티 해석과 활용 사례

#### 퍼플렉시티 해석 원칙

| 원칙 | 예시 |
|------|------|
| 구조화 데이터일수록 낮음 | HTML/코드 < 자연어 (`<head>` 다음엔 `</head>`) |
| 어휘 크기가 클수록 높음 | 아동 도서 < 복잡한 문학 < 전문 학술 문서 |
| 컨텍스트가 길수록 낮아짐 | 앞 내용이 많을수록 다음 토큰 예측 쉬워짐 |

#### 퍼플렉시티와 모델 품질의 관계

- **사전 학습 단계**: 퍼플렉시티가 모델 능력의 좋은 대리 지표
  - 모델 크기 증가(117M → 1542M 파라미터)에 따라 퍼플렉시티 감소 & 하위 작업 정확도 향상 (GPT-2)
- **사후 학습(SFT, RLHF) 이후**: 상관관계가 깨짐
  - 지시 따르기, 안전성 등을 최적화하면 순수 다음 토큰 예측 능력은 오히려 저하될 수 있음
  - 퀀타이제이션(quantization)도 퍼플렉시티에 예기치 않은 영향을 줄 수 있음

> ⚠️ 퍼플렉시티는 "언어 모델링 능력" 지표이지 "애플리케이션 품질" 지표는 아니다.  
> 퍼플렉시티가 낮다고 더 창의적이거나, 더 유용하거나, 사용자 만족도가 높은 것은 아니다.

#### 퍼플렉시티의 다양한 활용 사례

**1. 데이터 오염 탐지**
- 학습 데이터에 포함된 텍스트는 모델이 매우 낮은 퍼플렉시티를 보임
- 벤치마크 데이터의 퍼플렉시티가 비정상적으로 낮으면 오염 의심

**2. 데이터 다양성 확보**
- 새 데이터의 퍼플렉시티가 특정 임계값 이상일 때만 학습 데이터에 추가
- 퍼플렉시티가 너무 낮은 데이터(이미 많이 본 것)는 제외 → 중복 제거

**3. 이상 텍스트(Anomaly) 탐지**
- '강아지가 양자 물리학을 가르친다' 같은 비상식적 문장은 퍼플렉시티가 높음
- 스팸, 저품질 텍스트, 이상 입력 탐지에 활용 가능

## 3.3 정확한 평가 (Exact Evaluation)

확률 기반 지표(퍼플렉시티 등)는 충분하지 않다. 애플리케이션 수준 평가가 필요하다.

**정확한 평가** vs **주관적 평가**

| 구분 | 특징 | 예시 |
|------|------|------|
| 정확한 평가 | 명확하고 반복 가능한 판단 | 객관식 정답 = A, 모델 출력 = B → 단순히 틀림 |
| 주관적 평가 | 평가자·시기에 따라 달라짐 | 에세이 채점, AI 판사 (어떤 모델/프롬프트를 쓰느냐에 따라 다름) |

이 절에서는 **개방형 응답에 대한 정확한 평가**에 집중한다.

### 3.3.1 기능적 정확성 (Functional Correctness)

'모델이 실제로 문제를 해결했는가?'를 묻는 평가 방식.

- 웹사이트 생성 → 출력 HTML/JS가 명세를 만족하는가?
- 식당 예약 → 실제로 예약이 완료되었는가?

**애플리케이션에서 가장 궁극적인 지표**: 시스템이 올바르게 작동하지 않으면 다른 지표는 의미 없음

#### 코드 생성: 기능적 정확성의 최적 적용 사례

```python
# 문제 예시 (HumanEval)
def has_close_elements(numbers: List[float], threshold: float) -> bool:
    """주어진 숫자 리스트에서 threshold보다 가까운 두 수가 있는지 확인
    >>> has_close_elements([1.0, 2.0, 3.0], 0.5)
    False
    >>> has_close_elements([1.0, 2.8, 3.0, 4.0, 5.0, 2.0], 0.3)
    True
    """
    ...

# 테스트 케이스 (자동 채점)
def check(candidate):
    assert candidate([1.0, 2.0, 3.9, 4.0, 5.0, 2.2], 0.3) == True
    assert candidate([1.0, 2.0, 3.9, 4.0, 5.0, 2.2], 0.05) == False
```

#### 코드 생성 벤치마크

| 벤치마크 | 제공 | 평가 방식 |
|----------|------|-----------|
| HumanEval | OpenAI | 함수 시그니처+독스트링 → 테스트 케이스 통과 |
| MBPP | Google | 기초 파이썬 문제 |
| Spider | 예일대 | Text-to-SQL → 실제 DB 실행 결과 비교 |
| BIRD-SQL | - | Text-to-SQL (더 복잡한 쿼리) |

#### pass@k 지표

각 문제에 대해 $k$개의 코드 샘플 생성 후, 그 중 **하나라도 통과하면 성공**으로 간주한다.

$$\text{pass@k} = \frac{\text{k개 샘플 중 최소 1개 통과한 문제 수}}{\text{전체 문제 수}}$$

- 예: 10개 문제, $k=3$, 5개 문제를 3번 시도 중 최소 1번 통과 → pass@3 = 50%
- $k$가 커질수록 pass@k도 커짐 (더 많은 시도 = 더 많은 기회) → **테스트 시점 연산(test-time compute)** 효과

| 작업 | 성공 기준 |
|------|----------|
| 코드 생성 | 단위 테스트 통과 |
| SQL 생성 | DB 실행 결과 일치 |
| 게임 봇 | 달성 점수 |
| 최적화 작업 | 에너지 절감량 등 |

### 3.3.2 참조 데이터 유사도 측정

기능적 정확성을 자동화하기 어려운 경우, 참조 데이터(reference data)와 비교한다.

- 평가 데이터셋: `(입력, 참조 응답)` 형식
- 참조 데이터는 보통 사람이 작성하지만, 비용·속도 문제로 AI 생성 후 사람이 검토하는 방식도 증가

#### A. 정확 일치 (Exact Match)

출력이 참조와 문자 단위로 일치하는지 확인하는 이진(0/1) 지표.

- 적합한 경우: 수학 답(`5`), 퀴즈 정답(`파리`), 객관식
- **부분 일치 변형의 위험**:
  ```
  Q: 앤 프랭크가 태어난 연도?  (참조: 1929)
  모델 출력: "1929년 9월 12일"
  → '1929' 포함 → 통과
  → 그러나 실제 생일은 6월 12일 → 사실 오류
  ```
- 한계: 정답이 하나가 아닌 개방형 작업에는 적합하지 않음

#### B. 어휘 유사도 (Lexical Similarity)

표면적 단어/토큰 겹침을 측정한다 (의미는 고려하지 않음).

**N-gram 겹침**: N개 토큰 연속 시퀀스의 일치 비율

**편집 거리(Edit Distance)**: 텍스트 A를 B로 바꾸는 데 필요한 최소 연산 횟수
```
'bad' → 'bard': 삽입 1번 → 편집 거리 = 1
'bad' → 'cash': 연산 3번 → 편집 거리 = 3
```

**대표 지표**

| 지표 | 방향 | 주 용도 |
|------|------|---------|
| BLEU | N-gram 정밀도(Precision) | 기계 번역 |
| ROUGE | N-gram 재현율(Recall) | 요약 |
| METEOR | 정밀도+재현율+순서 | 번역 |

**한계**
- 어휘 겹침 ≠ 기능적 정확성
  - 잘못된 코드와 올바른 코드의 BLEU 점수가 비슷할 수 있음 (OpenAI 발견)
- 참조 편향(reference bias): 다른 단어로 맞는 답을 써도 낮은 점수
  - Adept의 Fuyu 모델: 의미적으로 동일한 정답이지만 참조와 표현이 달라 낮은 점수
- 현대 LLM 평가에서 BLEU/ROUGE는 번역 이외 작업에서는 **사실상 구식**

```
# 어휘 유사도 함정
"Let's eat, Grandma" vs "Let's eat Grandma"
→ 어휘적으로 99% 유사
→ 의미는 완전히 반대
```

#### C. 의미적 유사도 (Semantic Similarity)

단어가 달라도 **의미적 근접성**을 측정한다.

**임베딩(Embedding)**: 텍스트를 벡터 공간으로 변환  
**코사인 유사도(Cosine Similarity)**로 벡터 간 유사도 계산:

$$\text{sim}(A, B) = \frac{A \cdot B}{\|A\| \times \|B\|}$$

| 코사인 유사도 | 의미 |
|-------------|------|
| 1.0 | 동일한 방향 (거의 같은 의미) |
| 0.0 | 직각 (관련 없음) |
| -1.0 | 반대 방향 (반대 의미) |

```
"I like cats"  →  벡터 A
"I love felines"  →  벡터 B
→ 완전히 다른 단어지만 cosine similarity ≈ 0.9 이상
```

#### 유사도 지표 종합

| 지표 | 방식 | 대표 도구 | 주의사항 |
|------|------|-----------|----------|
| 정확 일치 | 문자열 동등 비교 | Python `==` | 형식 차이(5 vs 5.0)에도 실패 |
| 어휘 유사도 | 심볼 겹침 | BLEU(정밀도), ROUGE(재현율) | 단어가 다르면 맞는 답도 낮은 점수 |
| 의미적 유사도 | 벡터 거리 | BERTScore, Cosine Sim | 임베딩 모델 품질에 의존 |

**실전 계층화 접근법**

| 레이어 | 평가 방식 | 적합한 작업 |
|--------|----------|------------|
| Tier 1 | 정확 일치 | JSON/날짜 추출 |
| Tier 2 | 의미적 유사도 | RAG 검색 관련성 |
| Tier 3 | AI 판사 | 뉘앙스, 톤, 창의성 |

### 3.3.3 임베딩 소개

컴퓨터는 텍스트, 이미지, 오디오를 직접 처리할 수 없고 숫자만 이해한다.  
**임베딩**: 복잡한 비정형 데이터를 의미를 보존한 숫자 벡터로 변환하는 핵심 변환 레이어.

```
'고양이'와 '개' → 벡터 공간에서 가까운 위치
'고양이'와 '핵물리학' → 벡터 공간에서 먼 위치
```

#### 주요 임베딩 모델

| 제공사 | 모델명 | 차원 | 특징 |
|--------|--------|------|------|
| Google | BERT Base/Large | 768/1024 | 문장 수준 작업의 레거시 표준 |
| OpenAI | text-embedding-3-small | 1536 | 현 업계 표준, 효율 높음 |
| OpenAI | text-embedding-3-large | 3072 | 고정밀, 고비용 |
| Cohere | Embed v3 | 1024 | 검색/RAG 최적화 |
| OpenAI | CLIP | 512 | 멀티모달(텍스트↔이미지) |

#### 멀티모달 임베딩: CLIP과 그 이후

**기존 문제**: 텍스트 임베딩과 이미지 임베딩이 서로 다른 수학적 공간에 존재  
→ '개'라는 단어와 개 사진을 직접 비교할 수 없었음

**CLIP** (Contrastive Language-Image Pre-training, OpenAI)
- 텍스트와 이미지를 **같은** 임베딩 공간에 매핑
- 학습 방법: **대조 학습(Contrastive Learning)**
  - 텍스트-이미지 올바른 쌍(positive pair)의 유사도 → 최대화
  - 틀린 쌍(negative pair)의 유사도 → 최소화

```
이미지: [개 사진] → 벡터 A
텍스트: "a photo of a dog" → 벡터 B
→ A와 B의 dot product 최대화하도록 학습
```

**이후 발전**
- **ULIP**: 텍스트, 이미지, 3D 포인트 클라우드 통합
- **ImageBind**: 6가지 모달리티(텍스트, 이미지, 오디오, 깊이, 열화상, IMU) 통합

#### 임베딩 모델 선택 기준

- **MTEB(Massive Text Embedding Benchmark)** 리더보드 확인
- 차원 수 트레이드오프: 큰 벡터 → 더 세밀한 의미 표현, 더 높은 저장 비용 및 검색 지연
- Dense vs Sparse: 현대 임베딩은 Dense, 과거 TF-IDF는 Sparse
- 도메인 특화: 법률·의료 등 전문 도메인은 특화 임베딩 모델이 더 나을 수 있음

> 💡 **실전 팁**: RAG 시스템의 품질은 임베딩 모델에 크게 의존한다.  
> 'Apple'(과일)과 'Apple'(회사)를 컨텍스트 없이 같은 위치에 두는 모델은 검색 오류를 낸다.

## 3.4 AI 평가자 (AI as a Judge)

AI가 다른 AI의 출력을 평가하는 방식.

- GPT-3(2020) 이후 실용화되어 현재 가장 널리 쓰이는 평가 방식 중 하나
- LangChain 2023 보고서: 전체 평가의 **58%** 에서 AI 판사 활용

### 3.4.1 AI 평가자를 쓰는 이유

#### 장점

| 장점 | 설명 |
|------|------|
| 빠르고 저렴함 | 대규모 평가 자동화, 사람 평가 대비 비용 대폭 절감 |
| 참조 없이도 작동 | 프로덕션처럼 참조 데이터가 없어도 평가 가능 |
| 유연한 기준 | 정확성, 관련성, 충실성, 독성, 스타일 등 원하는 기준으로 평가 |
| 판단 근거 제공 | 판단 이유를 텍스트로 설명 → 디버깅·감사·실패 분석에 유용 |

#### 실증적 근거

- GPT-4와 인간의 MT-Bench 평가 일치율: 약 **85%** (인간 간 일치율 ~81%보다 높음)
- AlpacaEval 판사와 LMSYS Chatbot Arena(인간 투표 리더보드) 상관계수: ≈ **0.98**

> AI 판사는 완벽하지 않지만, '평가 없음'보다 압도적으로 낫고 대규모 사람 평가보다 훨씬 저렴하다.

### 3.4.2 AI 평가자 사용법

#### 패턴 1: 단일 답변 절대 품질 평가

질문 + 생성된 답변을 주고 점수 매기기.  
**사용 예**: 품질 대시보드, 모니터링, 실험 순위

```
다음 질문과 답변을 평가해 1~5점 사이로 점수를 매겨주세요.
- 1점: 매우 나쁨
- 5점: 매우 좋음

질문: [QUESTION]
답변: [ANSWER]
점수:
```

#### 패턴 2: 참조 답변과 비교 (유사도/정확도)

정답이 있을 때 생성 답변과 비교.  
BLEU/ROUGE 같은 정적 지표보다 의미적 유연성 제공.

```
다음 질문, 참조 답변, 생성된 답변을 보고
생성 답변이 참조 답변과 같은 내용인지 True/False로 출력하세요.

질문: [QUESTION]
참조 답변: [REFERENCE]
생성 답변: [GENERATED]
```

#### 패턴 3: 두 답변 쌍별 비교 (Pairwise Preference)

모델 순위 매기기, A/B 테스트, **RLHF/DPO용 선호도 데이터 생성**에 활용.

```
다음 질문과 두 가지 답변을 보고 어느 쪽이 더 좋은지 A 또는 B로 출력하세요.

질문: [QUESTION]
A: [FIRST ANSWER]
B: [SECOND ANSWER]

더 좋은 답변:
```

#### 주요 도구별 내장 기준

| 도구 | 내장 기준 |
|------|----------|
| Azure AI Studio | 기반성(groundedness), 관련성, 일관성, 유창성 |
| MLflow.metrics | 충실성(faithfulness 1~5), 관련성 |
| LangChain | 간결성, 정확성, 유해성, 일관성 등 |
| Ragas | 충실성(0 or 1), 답변 관련성 |
| LlamaIndex | 충실성(YES/NO) |

> ⚠️ 도구마다 같은 용어(예: 'faithfulness')의 정의와 점수 체계가 **다름** — 직접 비교 불가

#### 좋은 판사 프롬프트의 요소

1. **명확한 작업 정의**: 무엇을 평가할지
2. **구체적인 기준**: 세부적일수록 일관성 향상
3. **점수 체계**: 이산형 숫자(1~5)가 연속형(0~1)보다 LLM에게 다루기 쉬움  
   LLM은 텍스트 레이블을 미세한 숫자 척도보다 잘 다룸
4. **예시 포함**: 1점, 3점, 5점이 어떤 답변인지 예시를 주면 일관성 크게 향상

> AI 판사 = **모델 + 프롬프트 + 샘플링 설정**의 조합  
> 셋 중 하나라도 바뀌면 사실상 **다른 판사**를 쓰는 것

### 3.4.3 AI 평가자의 한계

#### 1) 비일관성

AI 판사는 확률적 모델이므로 동일 입력에도 점수가 달라질 수 있음.

- 완화 방법
  - 샘플링 파라미터 고정 (temperature = 0 또는 낮게)
  - 프롬프트에 예시 추가: 일관성 65% → 77.5%로 향상 (Zheng et al., 2023)
- 주의: 일관성이 높아도 **일관되게 틀릴** 수 있음

#### 2) 기준 모호성과 점수 비교 불가

판사 모델이나 프롬프트가 바뀌면 시간에 따른 점수 비교 불가.  
→ '오늘의 92%'와 '지난달의 90%'가 같은 의미인지 알 수 없음

**규칙**: 판사 모델과 프롬프트를 반드시 **버전 관리**해야 한다.

#### 3) 비용과 지연 시간

| 시나리오 | 호출 수 |
|---------|---------|
| 생성만 | 1×  |
| 생성 + 평가 | 2× |
| 생성 + 기준 3개 평가 | 4× |

- 완화 방법
  - 더 약하고 저렴한 모델을 판사로 활용
  - 스팟 체크: 전체 응답의 1~5%만 평가
  - 비동기 실행: 백그라운드에서 로깅 목적으로 실행

#### 4) 편향 (Bias)

| 편향 유형 | 설명 | 완화 방법 |
|----------|------|----------|
| 자기 편향(Self-preference) | 모델이 자신의 출력을 더 선호 (GPT-4: ~10%, Claude-v1: ~25%) | 다른 모델을 판사로 활용 |
| 위치 편향(Position bias) | A/B 비교에서 먼저 나온 것을 선호 | 순서 랜덤화, 두 번 비교 후 평균 |
| 장황함 편향(Verbosity bias) | 오류 있어도 더 긴 답변 선호 | 프롬프트에 길이 중립 지시 추가 |
| IP 우려 | 독점 판사 사용 시 데이터 외부 전송 | 온프레미스 모델 검토 |

### 3.4.4 평가자로 활용 가능한 모델

#### 구성 1: 평가 대상보다 강한 판사

- 채점자가 피험자보다 더 많이 알아야 한다는 직관
- 강한 모델(GPT-4 등)이 약한 모델의 일부(1%) 응답만 판단하는 비용 효율적 방식
- 한계: 가장 강한 모델에는 더 강한 판사가 없음 → 인간 평가나 앙상블 필요

#### 구성 2: 자기 평가 (Self-Critique)

```
사용자: 10 + 3은?
모델: 30

자기 비판: 이 답이 맞는가?
모델: 아니요, 정답은 13입니다.
```

자기 편향 우려가 있지만, 자기 수정과 품질 향상에 실용적으로 활용.

#### 구성 3: 평가 대상보다 약한 판사

- 좋은 노래를 알아보기 위해 가수가 될 필요는 없다는 논리
- **생성보다 판단이 더 쉬울 수 있음**
- 좁은 기준에서는 작고 특화된 판사가 매우 강할 수 있음

#### 특화된 AI 판사

| 유형 | 입력 | 출력 | 예시 |
|------|------|------|------|
| **보상 모델** | (프롬프트, 응답) | 스칼라 점수 (0~1) | Google Cappy (360M) |
| **참조 기반 판사** | (후보 응답, 참조 응답) | 유사도/품질 점수 | BLEURT [-2.5, 1.0], Prometheus (1~5) |
| **선호도 모델** | (프롬프트, 응답1, 응답2) | 어느 쪽이 선호될지 | PandaLM, JudgeLM |

**RLHF와 보상 모델의 연결**

```
보상 모델(Reward Model) = 사실상 AI 평가자

RLHF 과정:
1. 기본 LM 학습 (사전 학습)
2. 인간이 선호도 데이터 수집
3. Reward Model 학습: R(prompt, response) → 선호 점수
4. PPO 등으로 정책 최적화
```

> ⚠️ **Goodhart's Law**: 지표가 목표가 되면, 그 지표는 더 이상 좋은 지표가 아니다.  
> 보상 모델을 과도하게 최적화하면 **보상 해킹(Reward Hacking)** 발생 가능.

## 3.5 비교 평가를 통해 모델 순위 정하기

포인트와이즈(절대 점수) 평가의 한계: '7/10'이 무엇을 의미하는지 주관적이고 평가자마다 다르다.

**비교 평가(Comparative Evaluation)**: 두 모델을 나란히 놓고 어느 쪽이 더 나은지 선택.  
인간이 인지적으로 '이게 더 좋다'고 판단하기가 '이게 7.2/10이다'보다 훨씬 쉽다.

현재 업계 표준 **LMSYS Chatbot Arena**가 이 방식을 사용한다.

#### 비교 평가 사용 시 주의사항

**1. 사실 vs 선호도 혼동 금지**

```
Q: '휴대폰 전자파가 암을 유발하는가?'
모델 A: '예' (대중적 오해)
모델 B: '아니오' (과학적 사실)
→ 사용자가 A에 투표하면 미신을 학습하도록 훈련하는 꼴
```

규칙: 스타일·창의성에는 선호도 투표, 사실에는 기능적 정확성 평가를 사용한다.

**2. 평가자가 답을 모르는 질문 금지**

사용자가 복잡한 수학 문제를 AI에게 물어보는 이유는 **모르기 때문**이다.  
그 사용자에게 두 답변 중 어느 쪽이 더 좋은지 고르라고 하면 노이즈 데이터만 생성된다.  
→ AI가 '인턴'(사용자가 아는 일을 빠르게 처리)일 때만 비교 피드백을 수집한다.

### 3.5.1 비교 평가의 과제들

#### 확장성 문제: 이차함수적 증가

$N$개 모델을 비교하면 필요한 쌍의 수는 $\binom{N}{2} = \frac{N(N-1)}{2}$로 이차함수적 증가한다.

- LMSYS(2024년 1월): 57개 모델 → 약 244,000번 비교, 쌍당 평균 153번 → 통계적으로 얇음

#### 비이행성(Non-transitivity) 문제

대부분의 순위 알고리즘은 **이행성**을 가정한다: A > B, B > C → A > C

현실에서는:
```
모델 A: 수학에서 B를 이김
모델 B: 시에서 C를 이김
모델 C: 코딩에서 A를 이김
→ A > B > C 같은 단순 순위가 불안정
```

#### 품질 관리 문제

- LMSYS 프롬프트의 0.55% 이상이 단순히 '안녕' / 'Hi' → 단순 프롬프트는 복잡한 모델 간 차이를 가려냄
- **인센티브 오정렬**: 독성 답변을 웃겨서 투표 / 벤더가 자신의 모델에 과적합된 프롬프트 대량 제출

#### 순위를 계산하는 수학: Elo와 Bradley-Terry

**Elo 레이팅** (체스에서 유래)
- 레이팅 차이에 따라 승리 확률 계산
- 이긴 모델은 레이팅 상승, 진 모델은 하락
- 문제: 매치 순서에 민감, 불안정

```python
def elo_update(rating_a, rating_b, outcome, k=32):
    """
    outcome: 1 = A 승리, 0 = B 승리, 0.5 = 무승부
    """
    expected_a = 1 / (1 + 10 ** ((rating_b - rating_a) / 400))
    rating_a_new = rating_a + k * (outcome - expected_a)
    return rating_a_new

# 예시: A(1500) vs B(1600), A가 이겼을 때
new_rating = elo_update(1500, 1600, 1)
print(f"A의 새 레이팅: {new_rating:.1f}")  # 예상보다 큰 상승
```

**Bradley-Terry 모델** (LMSYS가 Elo에서 전환한 이유)
- 쌍별 비교용으로 설계된 확률 모델
- 순서 민감도 문제 해결

$$P(i \text{ beats } j) = \frac{\beta_i}{\beta_i + \beta_j}$$

- $\beta_i, \beta_j$: 각 모델의 능력 파라미터 (모든 매치 결과로부터 최대우도추정)

**TrueSkill** (Microsoft, Xbox Live에서 사용)
- 플레이어 능력에 대한 불확실성을 베이지안 방식으로 모델링

> 순위가 '맞다'고 볼 수 있으려면 **예측력**이 있어야 한다.  
> A가 B보다 순위가 높으면, A가 미래 대결에서 B를 50% 이상의 확률로 이겨야 한다.

#### 상대 순위 ≠ 절대 품질

| 시나리오 | 모델 A | 모델 B | 결론 |
|---------|--------|--------|------|
| 케이스 1 | 작업 성공률 0% | 작업 성공률 0% (B가 51% 승률) | B가 나은 쓰레기 |
| 케이스 2 | 작업 성공률 99% | 작업 성공률 99% (B가 51% 승률) | 사실상 동일 |

비교 평가는 '누가 더 나은가(상대)'를 알려주지, '충분히 좋은가(절대)'를 알려주지 않는다.  
B가 2배 비싸면서 51%만 이기면 전환이 합리적인가? → 비용 대비 가치 판단도 불가

### 3.5.2 비교 평가의 미래

#### 초인간 AI 시대에도 비교 평가가 필요한 이유

Llama 2 연구 팀의 인사이트:
- 모델이 인간 수준을 넘어서면 인간이 참조 답안을 **직접 작성**하는 것은 불가능해짐
- 하지만 두 답변을 나란히 보며 **어느 쪽이 더 좋은지 인식**하는 것은 여전히 가능

기존 벤치마크는 결국 포화되지만, 비교 평가는 항상 '더 나은' 모델이 있는 한 포화되지 않는다.

#### 실전 평가 전략

| 사용자 유형 | 방식 |
|-----------|------|
| 일반 사용자 | 👍/👎 단순 피드백 (포인트와이즈) |
| 내부 전문가 팀 | 나란히 비교 (비교 평가) |

**평가 계층 구조**
1. 코드/수학: 기능적 정확성 (단위 테스트) → 절대 기준
2. 사실/RAG: 정확 일치 / 의미적 유사도 → 참조 기반
3. 톤/창의성: Elo 기반 비교 평가 → 상대 기준

> 비교 승자 데이터는 고품질 학습 데이터 → 보상 모델 학습 및 미래 RLHF/DPO 파인튜닝에 활용 가능

| 모델 전반 비교 | 비교 평가 리더보드 | 전반적 품질 상대 순위 |
