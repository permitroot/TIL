## 3.2 언어 모델링 지표 이해하기
#### Q. LLM의 확률 예측 능력을 어떻게 수치화할 수 있는가?
- 정보이론 기반 지표인 교차 엔트로피와 퍼플렉시티를 사용한다
### 3.2.1 엔트로피 (Entropy)
#### Q. (정보이론의 핵심 개념) 정보량이란?
- 수식
  - $I(x) = -\log p(x)$
    - 이해
      - $-\log$는 "확률"을 "정보의 크기"로 바꿔주는 변환
        - 확률이 곱해질 때 → log는 더해진다
        - $\log(p)$는 $p$가 1보다 작으면 항상 음수
          - 정보량은 항상 양수 (0 이상)
    - 예시
      - 동전 두 번 던지기:
        - $p = 0.5 \times 0.5 = 0.25$
        - $-\log(0.25) = 2$
      - 한 번 정보량 = 1 bit, 두 번 정보량 = 2 bit
      - 즉, log는 "곱하기 확률"을 "더하기 정보"
- 의미
  - 확률이 낮은 사건 → 더 놀랍다 → 정보량 큼
  - 확률이 높은 사건 → 덜 놀랍다 → 정보량 작음
- 예시
  - 예시 1: 동전
    - 앞면 확률 = 0.5, 정보량 = $-\log_2(0.5) = 1$ bit
      - 앞면이 나오면 1비트의 정보 획득
  - 예시 2: 매우 드문 사건
    - 확률 = 0.01, $-\log_2(0.01) \approx 6.64$
      - 훨씬 많은 정보
#### Q. 엔트로피?
- 수식
  $$H(p) = -\sum_x p(x)\log p(x)$$
  - 해석
    - 각 사건의 정보량 × 그 사건이 일어날 확률 → 전부 더한 값
    - $H = \mathbb{E}[I(x)]$, 정보량의 기대값(평균)
  - 이해
    - LLM 에서
      - "엔트로피가 높다?" → 모델이 헷갈림 → 다음 단어 후보 많음
      - "엔트로피가 낮다?" → 모델이 확신이 있음
  - 예시
    - "대한민국의 수도는 __ ?" 서울: 0.95, 부산: 0.02, 대구: 0.01 → 엔트로피 낮음
    - "오늘 기분이 __ ?" 좋다, 나쁘다, 이상하다, 별로다, 신난다 → 엔트로피 높음
- 의미
  - 불확실성의 평균량 혹은 정보량의 평균
    - 엔트로피는 "평균적으로 얼마나 놀라는가?"를 수치로 표현한 것
    - 어떤 사건이 일어날 때 우리가 얻는 정보량의 평균
    - 모델이 다음 토큰을 얼마나 헷갈리는지에 대한 지표
  - "토큰이 평균적으로 얼마나 많은 정보를 담고 있는지" 측정
- 참고
  - 자연어는 본질적으로 엔트로피가 높음
    - 높다고 나쁜 게 아님 → 세상이 복잡하다는 뜻
#### Q. 언어의 엔트로피가 낮다?
- 언어의 토큰이 담고 있는 정보가 적다
- 다음에 올 것을 더 쉽게 예측할 수 있다
### 3.2.2 교차 엔트로피 (Cross Entropy)
#### Q. 교차 엔트로피는 무엇을 측정하는가?
- 실제 분포와 모델 예측 분포의 차이를 측정한다.
- 낮을수록 모델이 실제 데이터를 잘 예측한다.
#### Q. 교차 엔트로피 최소화는 왜 KL divergence 최소화와 동일한가?
- 수학적 설명
  $$H(p, q) = H(p) + D_{KL}(p \| q)$$
  - $H(p)$: 데이터 분포의 엔트로피 (상수)
  - $D_{KL}(p \| q)$: 모델 분포와 데이터 분포의 차이
  - 따라서, $H(p, q)$를 최소화하는 것은 $D_{KL}(p \| q)$를 최소화하는 것과 같다.
- 결론
  - LLM 학습이 "데이터 분포 모방" 이라는 걸 수학적으로 설명 가능
  - 퍼플렉시티가 단순 점수가 아니라 분포 근사 품질이라는 점 강조 가능
### 3.2.3 문자당 비트(BPC)와 바이트당 비트(BPB)
#### Q. BPC, BPB ?
- BPC: Bits per Character
- BPB: Bits per Byte
### Q. 왜 문자당/바이트당 비트를 사용하는가?
- 토크나이저가 달라도 공정하게 모델을 비교하기 위해 정규화
#### Q. 엔트로피, 교차 엔트로피의 단위 (비트)
- 언어 모델의 교차 엔트로피가 6비트라면, 각 토큰을 표현하는 데 6비트가 필요
#### Q. 문자 인코딩
- ASCII는 문자당 7비트, UTF-8은 문자당 8~32비트
#### Q. 더 표준화된 지표는?
- 표준화된 지표는 바이트당 비트(BPB)
#### Q. 교차 엔트로피는 언어 모델이 텍스트를 얼마나 효율적으로 압축할 수 있는지도 파악 가능
- 언어 모델의 BPB가 3.43이면 원본 1바이트(8비트)를 3.43비트로 표현할 수 있다는 뜻
- 언어 모델은 원본 학습 텍스트를 원래 크기의 절반 이하로 압축할 수 있다는 의미
#### Q. 왜 교차 엔트로피는 압축률과 동일한가?
- Shannon 정리:
  $$L \ge H(X)$$
  - 평균 코드 길이 $L$ ≥ 엔트로피 $H(X)$
- LLM의 BPB → 최적 압축률 근사
### 3.2.4 퍼플렉시티 (Perplexity)
#### Q. 퍼플렉시티?
- 정의
  - 언어 모델이 다음 토큰을 얼마나 헷갈려하는지(불확실성)를 나타내는 지표
    - 교차 엔트로피가 모델이 다음 토큰을 예측하기 얼마나 어려운지를 측정한다면, 퍼플렉시티는 다음 토큰을 예측할 때의 불확실성을 측정
  - 엔트로피와 교차 엔트로피의 지수 함수
- 의미
  - 모델이 평균적으로 몇 개의 선택지 중에서 고르는지를 나타낸다.
  - 낮을수록 더 자신감 있는 예측이다.
#### Q. 퍼플렉시티는 왜 교차 엔트로피의 지수인가?
- 정의
  $$\text{Perplexity} = 2^{H(p,\, q)}$$
  또는 자연로그 기준
  $$\text{PPL} = e^{H(p,\, q)}$$
- 핵심 의미
  - 퍼플렉시티는 "평균적으로 몇 개 선택지 중 고르는가"
  - 로그 공간(엔트로피) → 실제 선택 공간(지수화)
- 왜 중요?
  - 퍼플렉시티가 단순 점수가 아니라
  - 지수 스케일 변화라는 걸 설명할 수 있음
### 3.2.5 퍼플렉시티 해석과 활용 사례
#### Q. 퍼플렉시티가 낮으면 좋은 모델인가?
- 사전학습 단계에서는 그렇다.
- 그러나 사용자 체감 품질과는 직접적으로 일치하지 않는다.
#### Q. 퍼플렉시티를 해석할 때 사용할 수 있는 규칙들
- 구조화된 데이터일수록 퍼플렉시티가 낮음
- 어휘 크기가 클수록 퍼플렉시티가 높음
- 컨텍스트 길이가 길수록 퍼플렉시티가 낮음
#### Q. 활용 사례?
- 퍼플렉시티를 통해 비정상적인 텍스트를 탐지
  - 특이한 생각을 표현하는 텍스트나 의미 없는 텍스트처럼 예측하기 어려운 텍스트에서 가장 높게 나타남

## 3.3 정확한 평가 (Exact Evaluation)
#### Q. 확률 지표 외에 실제 성능을 어떻게 평가할 수 있는가?
- 기능적 정확성, 참조 기반 비교, 의미적 유사도 등을 사용
### 3.3.1 기능적 정확성 (Functional Correctness)
#### Q. 가장 신뢰할 수 있는 평가는 무엇인가?
- 실제로 문제를 해결했는지 테스트
  - 코드의 경우 단위 테스트 통과 여부
### 3.3.2 참조 데이터 유사도 측정
#### Q. 참조 답변과 비교하는 방법에는 무엇이 있는가?
- 정확 일치
- 어휘 유사도 (BLEU/ROUGE)
- 의미적 유사도 (임베딩 기반)
### 3.3.3 임베딩 소개
#### Q. 투영(projection)이란 무엇인가?
- 수학적으로:
  $$y = Wx$$
  - $x$: 입력 벡터, $W$: 투영 행렬, $y$: 변환된 벡터
  - 벡터를 다른 차원 또는 다른 표현 공간으로 바꾸는 것
- 딥러닝에서 벡터는 "좌표"
  - 하지만 문제는 텍스트 벡터 공간, 이미지 벡터 공간, Transformer 내부 hidden 공간이 각각 다릅니다.
  - 그래서 서로 다른 공간을 맞추기 위해 선형 변환이 필요합니다.
#### Q. 언어 투영행렬 (Language Projection Matrix)
- 예시
  - 단어 임베딩
    - 단어 → 300차원 벡터
    - Transformer hidden dimension → 768차원
  - 그럼?
    - $W_{proj} \in \mathbb{R}^{768 \times 300}$ 을 곱해서 차원 맞춤
  - 왜?
    - Transformer는 모든 토큰을 동일한 차원에서 처리
    - 차원이 안 맞으면 attention (어떤 단어를 얼마나 참고할지 가중치를 계산하는 과정) 계산 불가
#### Q. 비전 투영행렬 (Vision Projection Matrix)
- 예시
  - CLIP, GPT-4V, LLaVA 같은 모델
    - 이미지 인코더 (ViT 등) 출력: 1024차원
    - 언어 모델 hidden size: 4096차원
  - 그럼?
    - $W_{vision} \in \mathbb{R}^{4096 \times 1024}$
  - 텍스트 공간으로 변환
#### Q. 멀티모달 모델
- 핵심
  - 이미지와 텍스트가 같은 의미 공간에 존재 필요
    - 둘의 벡터가 가까워야 함
- 투영행렬은 학습을 통해 이미지 특징 → 언어 의미 공간으로 변환, 언어 특징 → 공통 의미 공간으로 변환
#### Q. Attention 내부에서도 투영 (Projection)
- Transformer에서 가장 중요한 부분
  $$Q = W_Q x, \quad K = W_K x, \quad V = W_V x$$
  - 여기서도 전부 "투영행렬"
- 왜?
  - 같은 입력이라도 역할이 다르기 때문
    - Query 역할
    - Key 역할
    - Value 역할
- 행렬은 서로 다르지만, 결과 차원은 맞춰야 한다.
#### Q. 임베딩은 평가에서 왜 중요한가?
- 표현이 달라도 의미적 유사성을 측정 가능
  - 특히 RAG 평가에서 핵심적

## 3.4 AI 평가자 (AI as a Judge)
#### Q. AI가 다른 AI를 평가해도 되는가?
- 가능하며 매우 효율적
  - 그러나 편향·비일관성·버전 관리 문제를 반드시 고려 필요

#### Q. 평가자의 예시
- 보상 모델
  - (프롬프트, 응답) 쌍을 입력 받고 주어진 프롬프트에 대해 그 응답이 얼마나 좋은지 점수를 매긴다.
  - 수년간 RLHF에서 사용되어 왔다.
  - ex) 구글의 캐피(cappy) - 3억 6천만 개의 파라미터를 가진 경량 평가 모델
- 참조 기반 평가자
  - 하나 이상의 참조 응답을 기준으로 생성된 응답을 평가한다.
  - 유사도 점수나 품질 점수를 출력할 수 있다.
  - ex) BLEURT, Prometheus
- 선호도 모델
  - (프롬프트, 응답1, 응답2)를 입력으로 받아 주어진 프롬프트에 대해 어느 응답이 더 나은지(사용자가 더 선호하는지) 출력한다.
  - 아마 특화된 평가자 중에서 가장 기대되는 방향일 것이다.
  - ex) PandaLM, JudgeLM
#### Q. 강한 모델이 반드시 좋은 판사인가?
- 반드시 그렇지 않음
  - 판단은 생성보다 쉬울 수 있으며, 특화된 작은 모델도 충분 할 수 있다.

## 3.5 비교 평가를 통해 모델 순위 정하기
#### Q. 모델 간 순위를 어떻게 정하는가?
- 절대 점수보다 **쌍별 비교(preference evaluation)**가 더 신뢰
#### Q. preference evaluation?
- 두 개의 모델 출력(A, B)을 나란히 보여주고, "어느 쪽이 더 좋은가?" 를 선택하게 하는 평가 방식
### 3.5.1 비교 평가의 과제들
#### Q. 쌍별 비교는 어떻게 수학적으로 순위화하는가?
- Bradley–Terry 모델
  - 정의
    $$P(i > j) = \frac{e^{s_i}}{e^{s_i} + e^{s_j}}$$
    - $s_i$: 모델 $i$의 잠재 점수
  - 왜 중요?
    - Chatbot Arena 같은 시스템의 기반 원리 설명 가능
    - "주관적 비교"가 사실은 확률 모델링임을 보여줌
### 3.5.2 비교 평가의 미래
#### Q. Llama(Large Language Model Meta AI)
- LLaMA는 Meta Platforms에서 개발한 대규모 언어 모델(LLM) 시리즈
  - 상대적으로 가볍고 효율적인 오픈 계열 LLM을 활용할 수 있도록 LLaMA를 공개
- 주요 특징
  - 고성능·고효율
    - 같은 규모의 다른 모델 대비 좋은 성능
    - 비교적 적은 자원으로도 구동 가능
  - 버전 발전
    - LLaMA → LLaMA 2 → LLaMA 3
      - 최신 버전일수록 추론 능력, 멀티모달 지원, 컨텍스트 길이 등이 개선

### 토론 질문
#### Q. 퍼플렉시티가 낮은데 사용자 만족도가 낮다면 어디서 문제가 생긴 것인가?
- 의견
  - 학습 목표와 제품 목표의 불일치에서 문제가 발생
- 설명
  - 퍼플렉시티는 "다음 토큰을 얼마나 잘 맞추는가?" 를 측정하는 언어 모델링 지표다.
  - 하지만 사용자는 여러가지를 원한다.
    - 정확한 정보
    - 명확한 구조
    - 안전한 답변
    - 의도 이해
    - 문제 해결
  - 즉,
    - 퍼플렉시티 ↓ = 확률 예측 능력 ↑, 사용자 만족도 ↓ = 실제 문제 해결 능력 부족
  - 특히 RLHF 이전 모델에서 자주 나타난다
- 결론
  - 퍼플렉시티는 학습 품질 지표, 사용자 만족은 제품 품질 지표다.
  - 두 지표는 직접적으로 일치하지 않는다.
#### Q. AI 평가자를 완전히 신뢰할 수 있을까? 어떤 조건에서 신뢰할 수 있는가?
- 의견
  - 조건부 신뢰는 가능하다.
- 이유
  - 왜 완전 신뢰 불가?
    - 자기 모델 편향 (self-preference bias)
    - 프롬프트 민감성
    - 버전 변경 시 점수 변동
    - 일관성 문제
- 신뢰 가능한 조건
  - 인간 샘플 검증으로 상관관계 확인
  - 판사 프롬프트 고정
  - 동일 판사 모델 유지
  - 절대 점수보다 쌍별 비교 사용
  - 주기적 인간 교차 평가
- 결론
  - AI 평가자는
    - "대규모 자동 필터링 도구"로는 신뢰 가능
    - "최종 품질 판정자"로는 인간 검증이 필요
#### Q. RAG 시스템에서 가장 중요한 평가 지표는 Retrieval인가, Generation인가?
- 의견
  - 1차적으로는 Retrieval이 더 중요
- 이유
  - RAG는 구조상:
    - 검색 → 문맥 구성 → 생성
  - 검색이 틀리면:
    - 생성은 아무리 잘해도 틀릴 가능성이 높다.
  - Retrieval Recall이 낮으면 정답이 문맥에 아예 존재하지 않는다.
- 결론
  - 1단계: Retrieval 품질 확보 (Recall 중심)
  - 2단계: Generation 정밀도 최적화
  - "Garbage in, garbage out" 검색이 먼저다.
#### Q. 코드 생성 모델에서 pass@1과 pass@10 중 어느 지표가 더 중요한가?
- 의견
  - 상황에 따라 다르다.
- 이유
  - 사용자 직접 사용하는 경우 (pass@1이 더 중요)
    - IDE 자동완성, ChatGPT 코드 답변 등
    - 사용자는 첫 답이 맞아야 한다
  - 자동 선택 시스템이 있는 경우 (pass@k(pass@10)가 더 중요)
    - 여러 개 생성, 테스트 통과 코드 자동 선택
    - 에이전트 구조에서는 샘플링 전략이 유효하다.
- 결론
  - 인터랙티브 제품 → pass@1
  - 자동 평가 파이프라인 → pass@k
#### Q. 초인간 AI 모델의 출력을 평가하기 위해 인간 평가자는 어떤 역할을 해야 하는가?
- 의견
  - 정답 생성자가 아니라 '선호 판단자' 역할을 해야 한다.
- 이유
  - 초인간 모델은:
    - 인간이 풀 수 없는 문제를 해결
    - 인간이 직접 정답을 만들 수 없음
  - 하지만 인간은 여전히 판단과 비교는 가능하다.
    - 논리적 일관성 판단
    - 유용성 판단
    - 안전성 판단
    - A vs B 비교
- 결론
  - 인간의 미래 역할은:
    - 절대 채점자 → ❌
    - 비교 평가자 → ✅
    - 정책·윤리 감독자 → ✅
    - 샘플 감사자(auditor) → ✅
